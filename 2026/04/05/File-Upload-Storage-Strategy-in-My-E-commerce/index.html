File Upload & Storage Strategy in My Backend Project

Background

When implementing file upload functionality, I initially followed a tutorial and directly integrated with Alibaba Cloud OSS. The controller handled file uploads and returned a full URL after storing the file in OSS.

However, during development, several issues became apparent:

  • The system could not run locally without cloud credentials
  • Business logic was tightly coupled with a specific storage provider
  • Switching storage solutions (OSS → S3 → MinIO) required code changes
  • Local file serving with Nginx introduced unnecessary complexity (e.g., SPA fallback issues)

To address these problems, I redesigned the file upload module.

Design Goals

  • Enable local development without external dependencies
  • Decouple storage implementation from business logic
  • Support multiple storage backends (Local / MinIO / S3)
  • Improve maintainability and extensibility

Key Idea

Treat storage as an external dependency and abstract it behind an interface.

In this design:

  • The upload API only handles file input
  • Storage is delegated to interchangeable implementations
  • The system returns a resource identifier (URL or object key)

Cloud Storage Implementation (Production)

Why Not Use OSS SDK Directly in Controller?

❌ Problematic Design

`controller → AliOssUtil → OSS SDK

  • Tight coupling with infrastructure
  • Hard to test locally
  • Difficult to switch providers

✅ Improved Design

`controller → StorageService → implementation

Benefits

  • Loose coupling
  • Better testability
  • Easy to extend and maintain

Key Insight: Resource Ownership

A critical design decision is:

The upload path should not be provided by the client.

Reasons

  • The path is a system-generated identifier (similar to a primary key)
  • Prevents conflicts and security issues
  • Allows flexible storage strategies

Environment Strategy

Environment Storage
Development Local filesystem / mock
Production S3 / OSS / MinIO

Lessons Learned

1. WSL vs Windows File System

Running the application in WSL while accessing files from Windows can lead to path inconsistencies.

2. Nginx Static Resource Conflicts

SPA configurations (e.g., try_files) may override /static/* routes, returning index.html instead of actual files.

3. Returning URL vs Object Key

In production systems, it is often better to:

  • Store and return an object key
  • Generate a pre-signed URL when access is needed

Summary

The file upload module evolved from:

❌ A tightly coupled, provider-specific implementation
➡️ To
✅ A flexible, interface-driven storage strategy

This design:

  • Enables local development without cloud dependencies
  • Decouples infrastructure from business logic
  • Simplifies switching between storage providers

Future Improvements

  • Pre-signed URL for private access
  • CDN integration
  • Image processing (compression, resizing)

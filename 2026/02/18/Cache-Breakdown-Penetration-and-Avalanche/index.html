Cache Breakdown, Penetration and Avalanche

缓存击穿、穿透与雪崩 —— 电商项目中的实践与思考

Introduction | 引言

EN

In e-commerce systems, cache is not just a performance optimization technique.
More importantly, it protects the database layer under traffic spikes and prevents cascading failures.

In my personal e-commerce project, I introduced Redis to practice production-oriented design:
concurrency control, failure-awareness, and trade-offs.
This article focuses on real scenarios rather than Redis installation or basic CRUD.

CN

在电商系统中，缓存不仅是性能优化手段，更重要的是在流量波动下保护数据库层，避免级联故障。

在我的个人电商项目中，我引入 Redis 的目标是训练生产化思维：并发控制、故障意识以及设计权衡。
本文不讲 Redis 安装和基础用法，而是从真实业务场景出发，总结三类典型缓存问题及应对策略。

0. Quick Definitions | 三种常见的缓存问题定义

EN

  • Penetration: the key does not exist (DB also has no record) → every request hits DB
  • Breakdown / Hot key: a hot key expires → many concurrent requests hit DB at once
  • Avalanche: many keys expire around the same time → traffic spikes hit DB together

CN

  • **穿透（penetration）：请求的 key 根本不存在（DB 也没有），每次都打到 DB
  • **击穿（breakdown / hot key）：某个热点 key 过期瞬间，大量请求同时打 DB
  • **雪崩（avalanche）：大量 key 同一时间过期，流量一起打 DB

1. Cache Penetration | 缓存穿透（不存在的 key 反复查询）

Scenario | 场景

EN

A user (maliciously or accidentally) repeatedly requests a non-existent product id, such as id=99999999.
Since the record does not exist in DB, the cache never gets populated, and every request reaches DB.

CN

典型场景：用户恶意/误请求不存在的商品 id，例如 id=99999999。
由于 DB 根本不存在该记录，缓存不会被填充，导致请求每次都打到 DB。

Impact | 后果

EN

  • DB QPS spikes
  • Cache becomes ineffective
  • Can be amplified by scanning/attack traffic

CN

  • DB QPS 激增
  • Redis 保护层形同虚设
  • 恶意扫描下风险会被放大

Solutions | 解决方案

EN

  1. Input validation (cheapest)
  2. Negative cache (cache null with short TTL)
  3. Bloom filter (when data is large & traffic is high)

CN

  1. 参数校验（最便宜）
  2. 缓存空值（negative cache）：DB 查不到也缓存占位符，TTL 短一点
  3. Bloom Filter（数据量大、访问高时）

My Choice | 我的选择：Negative Cache

EN

Given my project scale, negative caching provides the best ROI.
When DB returns null, I cache a placeholder with a short TTL to stop repeated DB hits.

CN

由于个人项目并不具备真实生产级高并发规模，我选择 ROI 更高的 缓存空值。
当 DB 查询为 null 时，写入一个占位符并设置较短 TTL，从而阻断重复穿透。

2. Cache Breakdown | 缓存击穿：热点 key 过期瞬间并发打 DB

Scenario | 场景

EN

A hot key (e.g., popular dish/product detail) expires.
At that moment, a burst of concurrent requests bypasses cache and hits DB simultaneously.

CN

热点 key（例如爆款商品详情）在某个时刻过期，
大量并发请求在缓存回填之前同时访问 DB，造成瞬时压力激增。

Solutions | 解决方案

EN

  1. Mutex lock (SETNX): only one request rebuilds cache
  2. Logical expiration: never “hard expire”, refresh asynchronously
  3. Never expire + manual refresh: for a small number of hot keys

CN

  1. 互斥锁（SETNX）：只让一个请求去 DB，其余等待回填
  2. 逻辑过期（logical expire）：key 不“硬过期”，后台线程异步刷新
  3. 热点 key 永不过期 + 手动刷新（热点少时）

My Choice | 我的选择：Mutex Lock + Double Check

EN

I chose mutex locking because it directly solves the root cause:
concurrent cache rebuild under hot-key expiration.

Flow:

  • cache miss → try lock
  • lock holder queries DB and fills cache
  • others wait briefly and retry
  • double-check avoids redundant DB queries

CN

我选择互斥锁，因为它最直接地对应问题本质：热点过期引发的并发重建。
流程是：

  • cache miss → 抢锁
  • 抢到锁的访问 DB 并回填
  • 其他请求短暂等待后重试
  • double-check 避免重复打 DB

3. Cache Avalanche | 缓存雪崩：大量 key 同时过期

Scenario | 场景

EN

Many keys share similar TTLs and expire around the same time.
A synchronized expiration wave causes a burst of DB traffic.

CN

大量 key TTL 接近，在同一时间段集中失效，
导致 DB 被瞬间流量冲击。

Solutions | 解决方案

EN

  1. Add random jitter to TTL
  2. Pre-warm / layered cache
  3. Rate limit + degrade gracefully

CN

  1. TTL 加随机抖动
  2. 预热 / 分层缓存
  3. 限流 + 降级

My Choice | 我的选择：TTL Jitter

EN

Given my project scale, TTL jitter is the simplest and most cost-effective approach.
It prevents synchronized expiration and smooths spikes. And this thinking is also used in algorithm and system design like DB sharding.

CN

考虑到项目规模与维护成本，TTL 随机抖动是最轻量且有效的方案，
用于打散过期时间，平滑流量峰值。同时这种散列思想也在分库分表和算法里有普遍应用。

Refection

1) Cache dependency risk | 缓存依赖风险

EN

A cache layer can become a single point of failure if business logic depends on it too heavily.
If Redis is down, features like shop status reading or JWT session validation may fail.

CN
缓存层如果被业务强依赖，会反过来成为单点故障。
例如 Redis 宕机后，店铺营业状态读取、JWT 登录鉴权等功能可能直接不可用。

Cache should enhance availability, not reduce it.
缓存应提升可用性，而不是降低可用性。

2) Consistency problems | 数据一致性

EN

Invalidation is harder than insertion.
When deleting data from DB, it’s easy to miss related cache keys (bundles, carts, historical orders), causing stale data.

CN

缓存失效（删除）往往比写入更难。
删除 DB 数据时，极易遗漏关联对象（套餐、购物车、历史订单中的菜品引用等），导致脏数据长期存在。

3) Architectural direction | 架构方向

EN

Instead of scattering try-catch blocks everywhere, a cleaner approach is to encapsulate cache logic via a centralized abstraction (e.g., AOP). About AOP, I will introduce how how use it in my another post.

CN

与其在每个调用点写 try-catch，不如通过统一抽象层（例如 AOP）封装缓存访问与降级逻辑，提高一致性与可维护性。我会在其他推送里重点介绍我的EC项目里用到的AOP方法。
